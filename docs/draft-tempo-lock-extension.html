Tempo Payment Lock Extension

Abstract

This document extends the Tempo Payment Method [draft-tempo-payment-method] with support for escrowed payment locks. It defines the lock intent for incremental payments suitable for streaming and metered services, where clients deposit funds into an on-chain escrow that permits the payee to withdraw funds up to the amount authorized by signed releases, while permitting the payer to withdraw any remaining funds after expiry.В¶

1. Introduction

The base Tempo Payment Method [draft-tempo-payment-method] supports one-time charges, authorizations, and subscriptions. However, these intents require the payment amount to be known upfront or bounded by a fixed authorization.В¶

For metered servicesвҖ”where consumption is only known as the service is deliveredвҖ”a different model is required: one where clients deposit funds into escrow, receive service, and authorize incremental releases as they consume.В¶

This extension defines the lock intent, which establishes an escrowed payment lock using:В¶

  • TIP20Escrow interface for on-chain fund escrow (implementable as aВ¶

smart contract or chain-native precompile)В¶

  • Off-chain EIP-712 releases for incremental release authorizationВ¶
  • Payee-controlled settlement for efficient fund collectionВ¶

1.1. Example Use Case: Metered Agent Execution

This section is non-normative.В¶

Consider a user delegating tasks to an AI agent that consumes API tokens:В¶

  1. User initiates an agent task (e.g., a research task)В¶
  2. Agent's API returns 402 with a lock challengeВ¶
  3. User creates a lock with a budget for the taskВ¶
  4. As tokens are consumed, the client sends EIP-712 releases authorizing paymentВ¶
  5. Server settles periodically or when the task completesВ¶

The user pays for tokens consumed, with unused budget reclaimable after lock expiry. This enables users to delegate open-ended tasks to agents with bounded spending.В¶

1.2. Trust Model and Fund Commitment

In this document, the server acts as the payee, and the client acts on behalf of the payer.В¶

Payment locks use an escrow-based model with on-chain fund locking:В¶

  1. Funds are locked at creation. The payer deposits TIP-20 tokens intoВ¶

a TIP20Escrow implementation. These funds are locked until release or expiry.В¶

  1. Payee can release at any time. Using a signed release from the payer,В¶

the payee can withdraw funds from the escrow (up to the released amount).В¶

  1. Partial release is supported. The payee can release multiple times,В¶

each time withdrawing the delta between the release amount and previously released amount.В¶

  1. Payer can withdraw after expiry. If the payee fails to release beforeВ¶

the lock expires, the payer can withdraw all remaining (unreleased) funds.В¶

  1. Payee bears the risk of not settling. The only risk to the payee isВ¶

failing to release before expiry; escrowed funds are committed for payment.В¶

Trust properties:В¶

  • Payer protection: Funds can only be released by payee with a validВ¶

signed release; after expiry, payer reclaims unreleased funds.В¶

  • Payee protection: Escrowed funds are committed for payment for validВ¶

releases (no risk of payer draining wallet).В¶

Cumulative release semantics:В¶

Releases MUST authorize a cumulative total, not incremental deltas. A release's cumulativeAmount MUST be strictly greater than the previously accepted cumulativeAmount for the same lock.В¶

  • Release #1: cumulativeAmount = 100000 (authorizes 100000 total)В¶
  • Release #2: cumulativeAmount = 250000 (authorizes 250000 total)В¶
  • Release #3: cumulativeAmount = 400000 (authorizes 400000 total)В¶

When settling, the implementation computes: delta = cumulativeAmount - releasedВ¶

This permits the lock to remain usable after partial releasesвҖ”the payer continues signing releases with higher cumulative amounts up to the deposit.В¶

3. Terminology

Payment Lock

An on-chain escrow where the payer deposits TIP-20 tokens that can only be released to a designated payee using signed releases, or withdrawn by the payer after expiry.В¶

Release

An EIP-712 signed message authorizing a cumulative release amount for a specific lock. Releases are monotonically increasing in amount.В¶

TIP20Escrow

An interface for payment lock escrow, providing lock, release, requestRelease, and withdraw operations. Implementations MAY be deployed as smart contracts or chain-native precompiles.В¶

Lock ID

A unique identifier (bytes32) for a specific lock, returned by a TIP20Escrow implementation when a lock is created.В¶

Cumulative Amount

The total amount authorized for release since lock creation. Each release's cumulativeAmount MUST be greater than the previous.В¶

4. Intent: "lock"

An escrowed payment lock for streaming or metered services. The payer deposits funds into an on-chain escrow (a TIP20Escrow implementation) that permits the payee to withdraw funds up to the amount authorized by releases, while permitting the payer to withdraw any remaining funds after expiry.В¶

A lock request MUST include the parameters defined as REQUIRED in Section 5. A server challenge for intent="lock" MUST NOT omit any REQUIRED fields.В¶

5. Lock Request Schema

For intent="lock", the request parameter contains a base64url-encoded JSON object with the following fields:В¶

Table 1
Field Type Required Description
asset string REQUIRED TIP-20 token address
amount string REQUIRED Amount to lock in base units (stringified number)
expiry string REQUIRED Lock expiry as UNIX timestamp in seconds (stringified number)
payee string REQUIRED Recipient address for released funds
trust string OPTIONAL Exclusive intermediary authorized to call release (if set, payee cannot)
feePayer boolean OPTIONAL If true, server will pay transaction fees (default: false)

If trust is present, only trust can call release(); the payee cannot. Funds are still transferred to payee, but trust acts as an exclusive intermediary (e.g., an arbitration service). If trust is absent, payee calls release() directly.В¶

The following is a non-normative example:В¶

{
  "asset": "0x20c0000000000000000000000000000000000001",
  "amount": "10000000",
  "expiry": "1737043200",
  "payee": "0x742d35Cc6634C0532925a3b844Bc9e7595f8fE00"
}
В¶

In this example, the request specifies a lock of 10.00 alphaUSD (10000000 base units) expiring at UNIX timestamp 1737043200.В¶

The client MUST fulfill this by signing a Tempo Transaction that:В¶

  1. Calls lock(asset, amount, expiry, payee, trust) on a TIP20EscrowВ¶

implementationВ¶

  1. The implementation transfers amount of asset from the payer to escrowВ¶
  2. Returns a lockId that identifies this specific lockВ¶

The trust field, if present, specifies an additional address that can release funds (e.g., an arbitration service). If omitted, only the payee can release funds before expiry.В¶

7. EIP-712 Release Format

To release funds from a lock, the payee MUST provide an EIP-712 signed release from the lock owner. The signed release authorizes the payee to withdraw funds up to cumulativeAmount from the lock.В¶

7.1. Domain Separator

The EIP-712 domain MUST use the following values:В¶

Table 7
Field Value
name "TIP20Escrow"
version "1"
chainId Tempo chain ID (e.g., 42431)
verifyingContract Address of the TIP20Escrow implementation that created the lock

7.2. Release Type 

struct Release {
    bytes32 lockId;
    uint256 cumulativeAmount;
    uint256 expires;
}
В¶
Table 8
Field Type Description
lockId bytes32 Lock identifier
cumulativeAmount uint256 Total amount authorized for release (cumulative, not delta)
expires uint256 UNIX timestamp after which this release expires

7.3. Cumulative Semantics

Releases MUST authorize a cumulative total, not incremental deltas. A TIP20Escrow implementation MUST reject any release where cumulativeAmount <= released. A payee SHOULD use the highest valid release available when calling release().В¶

  • Release #1: cumulativeAmount = 100000 вҶ’ releases 100000В¶
  • Release #2: cumulativeAmount = 250000 вҶ’ releases 150000 more (delta)В¶
  • Release #3: cumulativeAmount = 400000 вҶ’ releases 150000 more (delta)В¶

The implementation computes: delta = cumulativeAmount - lock.releasedВ¶

This prevents replay attacks and permits the lock to remain usable after partial releasesвҖ”the payer continues signing releases with higher cumulative amounts up to the locked deposit.В¶

8. TIP20Escrow Interface

The TIP20Escrow interface defines the escrow functionality for payment locks. Implementations MAY be deployed as:В¶

  • Smart contracts: Standard EVM contracts deployable on any EVM chainВ¶
  • Precompiles: Chain-native implementations at fixed addresses forВ¶

improved gas efficiencyВ¶

The interface is described using Solidity for exposition; non-Solidity implementations MUST provide equivalent semantics.В¶

8.1. Lock State

Each lock is identified by a unique lockId and stores:В¶

Table 9
Field Type Description
active bool Whether the lock is active
asset address TIP-20 token address
amount uint256 Total amount deposited
released uint256 Cumulative amount already released
expires uint256 UNIX timestamp after which payer can withdraw
payee address Recipient of released funds
trust address If set, exclusive caller for release (payee cannot call)

8.2. Functions

8.2.1. lock

Creates a new payment lock with escrowed funds.В¶

function lock(
    address asset,
    uint256 amount,
    uint256 duration,
    address payee,
    address trust
) external returns (bytes32 lockId);
В¶

The implementation MUST:В¶

  1. Transfer amount of asset from msg.sender to the implementationВ¶
  2. Compute expires = block.timestamp + durationВ¶
  3. Generate unique lockIdВ¶
  4. Store lock state with released = 0В¶
  5. Emit LockCreated eventВ¶
  6. Return lockIdВ¶

8.2.2. release

Payee releases funds using a signed release.В¶

function release(
    address account,
    bytes32 lockId,
    uint256 cumulativeAmount,
    bytes calldata signature
) external;
В¶

The implementation MUST:В¶

  1. If lock.trust is set: verify msg.sender == lock.trustВ¶

If lock.trust is not set: verify msg.sender == lock.payeeВ¶

  1. Verify block.timestamp < lock.expiresВ¶
  2. Verify EIP-712 signature from account (lock owner)В¶
  3. Verify cumulativeAmount > lock.released (monotonicity)В¶
  4. Verify cumulativeAmount <= lock.amount (cap)В¶
  5. Compute delta = cumulativeAmount - lock.releasedВ¶
  6. Transfer delta to lock.payeeВ¶
  7. Update lock.released = cumulativeAmountВ¶
  8. Emit FundsReleased eventВ¶

The implementation MUST reject releases after expiry.В¶

8.2.3. requestRelease

Payer requests early release of the lock.В¶

function requestRelease(bytes32 lockId) external;
В¶

The implementation MUST:В¶

  1. Verify msg.sender is lock ownerВ¶
  2. Set lock.releaseRequested = block.timestampВ¶
  3. Emit ReleaseRequested eventВ¶

After the implementation-defined grace period, the payer can call withdraw(). The grace period SHOULD be documented by the implementation.В¶

8.2.4. withdraw

Payer withdraws remaining funds after expiry or grace period.В¶

function withdraw(bytes32 lockId) external;
В¶

The implementation MUST:В¶

  1. Verify msg.sender is lock ownerВ¶
  2. Verify either:В¶

- block.timestamp >= lock.expires, OR - lock.releaseRequested > 0 && block.timestamp >= lock.releaseRequested + gracePeriodВ¶

  1. Compute remaining = lock.amount - lock.releasedВ¶
  2. Transfer remaining to lock ownerВ¶
  3. Mark lock as inactiveВ¶
  4. Emit LockClosed eventВ¶

8.2.5. modifyLock

Payer adds funds and/or extends expiry on an existing lock.В¶

function modifyLock(
    bytes32 lockId,
    uint256 additionalAmount,
    uint256 newExpiry
) external;
В¶

The implementation MUST:В¶

  1. Verify msg.sender is lock ownerВ¶
  2. Verify lock is activeВ¶
  3. If additionalAmount > 0:В¶

- Transfer additionalAmount of the lock's asset from payer to escrow - Add to lock.amountВ¶

  1. If newExpiry > lock.expires:В¶

- Update lock.expires = newExpiryВ¶

  1. Emit LockModified eventВ¶

The implementation MUST reject if newExpiry < lock.expires (expiry can only be extended, not shortened).В¶

function getLock(
    address account,

9. Verification Procedure

9.1. Lock Verification

For intent="lock" credentials, servers MUST verify the lock state on-chain before granting access:В¶

  1. Extract lockId from the credential payloadВ¶
  2. Call getLock(payer, lockId) on the TIP20Escrow implementationВ¶
  3. Verify the returned lock state:В¶

- active is true - asset matches the requested token - amount >= the requested minimum amount - expires MUST be greater than the current time. Servers SHOULD enforce an implementation-defined minimum remaining lifetime suitable for the requested service. - payee matches the server's addressВ¶

  1. If trust was specified in the request, verify it matchesВ¶

Servers SHOULD cache lock state and periodically re-verify for long-running services.В¶

9.2. Release Verification

Servers receiving a type="release" credential MUST perform the following checks:В¶

  1. Verify id matches an active lock sessionВ¶
  2. Extract release fields: lockId, cumulativeAmount, expires, signatureВ¶
  3. Verify cumulativeAmount > previously received release amount (monotonicity)В¶
  4. Verify expires > current timeВ¶
  5. Verify EIP-712 signature matches the lock owner (see Section 7)В¶
  6. Store the release for later settlementВ¶

Servers SHOULD acknowledge receipt of valid releases. Servers MAY batch multiple releases and settle periodically rather than per-release.В¶

The following is an illustrative (non-normative) flow:В¶

  1. Client signs and broadcasts a transaction with lock() callВ¶
  2. Lock is created; client receives lockIdВ¶
  3. Client sends lock credential via Authorization: Payment headerВ¶
  4. Server verifies lock state on-chain and begins serviceВ¶
  5. As service streams, client signs EIP-712 releasesВ¶
  6. Client sends release credentials via Authorization: Payment headerВ¶
  7. Server collects releases; settles periodicallyВ¶
  8. Server calls release(payer, lockId, amount, signature) with latest releaseВ¶
  9. Implementation verifies signature and transfers delta to payeeВ¶

10.2. Post-Expiry Recovery

After expiry, the lock owner can reclaim all unreleased funds by calling withdraw(lockId) (no signature requiredвҖ”only the lock owner can withdraw their own funds after expiry).В¶

10.3. Early Release

If the payer wants to end the lock before expiry:В¶

  1. Payer calls requestRelease(lockId) on the TIP20Escrow implementationВ¶
  2. The implementation-defined grace period beginsВ¶
  3. Payee can still call release() with any pending releases during graceВ¶
  4. After grace period, payer can call withdraw(lockId) to reclaim remaining fundsВ¶

This mechanism permits the payee to settle outstanding authorized amounts during the grace period while permitting the payer to recover unspent funds thereafter.В¶

11. Security Considerations

11.1. Lock Security

Implementations MUST consider the following security properties:В¶

Expiry Management: Servers MUST release funds before the lock expires. After expiry, the payer can reclaim all remaining funds. Servers SHOULD:В¶

  • Set alerts for approaching lock expiryВ¶
  • Release funds well before expiry (with margin for transaction confirmation)В¶
  • Track all active locks and their expiry timestampsВ¶

Payee Address Verification: Clients MUST verify the payee address in the lock request matches the expected server. A malicious server could specify a different address to redirect funds.В¶

Lock Amount: Clients SHOULD limit locked amounts to those consistent with their risk tolerance. While expiry protects against indefinite fund locking, funds remain unavailable until expiry.В¶

Trust Address: If the trust field is specified, only that address can call release(); the payee cannot. Clients SHOULD only accept trusted arbitration services and verify the trust address is legitimate.В¶

Partial Release: TIP20Escrow implementations support partial releases. Servers SHOULD release only the amount consumed to minimize client exposure. Clients can reclaim unreleased funds after expiry.В¶

Early Release (requestRelease): Payers can request early release before expiry. This starts an implementation-defined grace period during which:В¶

  • The payee can still call release() with valid releasesВ¶
  • After the grace period, the payer can call withdraw() to reclaim fundsВ¶
  • Servers SHOULD monitor for ReleaseRequested events and settle promptlyВ¶

This permits payer flexibility while providing the payee time to finalize settlement.В¶

11.2. Release Security

Implementations MUST consider the following EIP-712 release security properties:В¶

Replay Prevention: Releases use cumulative semanticsвҖ”each release authorizes a total amount, not a delta. The implementation enforces:В¶

  • cumulativeAmount > lock.released (monotonicity)В¶
  • cumulativeAmount <= lock.amount (cap)В¶

This prevents replay: submitting the same release twice has no effect after the first settlement.В¶

Release Expiry: The expires field limits release validity. Clients SHOULD use short windows during active streaming to limit exposure if a release is intercepted. Implementations MUST reject releases where block.timestamp > expires.В¶

Signature Verification: Implementations MUST:В¶

  • Use strict ECDSA recovery to derive signer addressВ¶
  • Verify signer matches the lock owner (account parameter)В¶
  • Prevent signature malleability attacksВ¶
  • Bind signature to lockId, cumulativeAmount, and expiresВ¶

Release Interception: Even if an attacker intercepts a release:В¶

  • They cannot submit it (only payee/trust can call release())В¶
  • They cannot modify the amount (signature verification fails)В¶
  • They cannot use it on another lock (bound to lockId)В¶

Off-chain Release Delivery: Clients send releases to servers out of band (for example, via HTTP POST to a server-defined endpoint). The release credential includes both the EIP-712 message and signature.В¶

When release credentials are transmitted over HTTP, the client MUST use TLS [RFC8446]. Servers SHOULD acknowledge receipt of valid releases.В¶
